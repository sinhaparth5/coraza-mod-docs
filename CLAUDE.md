## Commands

```bash
npm run start        # Dev server at http://localhost:3000 (hot reload)
npm run build        # Static build into build/
npm run serve        # Serve the build/ output locally
npm run clear        # Clear Docusaurus cache (.docusaurus/ and build/)
npm run typecheck    # TypeScript type-check (tsconfig.json)
npm run deploy       # Build and push to gh-pages branch
```

Node.js >=20.0 is required.

## Architecture

This is a **Docusaurus v3.10.1** documentation site for **Coraza WAF Mod** (a WAF project hosted on GitLab, not GitHub). It uses the `classic` preset with `@docusaurus/faster` enabled (Rspack-based bundler for faster builds). The `docs/` and `blog/` content is still the default Docusaurus tutorial scaffolding — the custom work so far is in the homepage, the roadmap page, and the swizzled sidebar.

Note a deploy mismatch: source/issues/`editUrl` point to GitLab (`sinhaparth5/coraza-waf-mod`), but `npm run deploy` pushes to a `gh-pages` branch and `url` in the config is still the placeholder `your-docusaurus-site.example.com`. Confirm the intended host before relying on `deploy`.

### Key files

- `docusaurus.config.ts` — site-wide config: title, URL, navbar, footer, plugins, theme. `organizationName`/`projectName`/`editUrl` are set to GitLab values; `url` is still a placeholder.
- `sidebars.ts` — sidebar config. Currently uses `autogenerated` (filesystem-based ordering via `sidebar_position` frontmatter). Switch to explicit config here when manual ordering is needed.
- `src/pages/` — standalone pages (not part of the docs sidebar). Each `.tsx` or `.mdx` file becomes a route. `index.tsx` is the custom homepage; `roadmap.tsx` is the live roadmap.
- `src/css/custom.css` — global CSS and Docusaurus CSS variable overrides (color scheme, fonts, etc.). Fonts (Plus Jakarta Sans, JetBrains Mono) are loaded via `stylesheets` in the config.
- `docs/` — all documentation content as `.md` or `.mdx`. Sidebar order is controlled by `sidebar_position` in each file's frontmatter.
- `blog/` — blog posts. Authors defined in `blog/authors.yml`, tags in `blog/tags.yml`.
- `static/` — files copied verbatim to `build/` root (images, favicons, etc.).

### Custom code (the non-scaffolding parts)

- **Swizzled sidebar** — `src/theme/DocSidebar/Desktop/Content/index.tsx` wraps the original Docusaurus sidebar content to render a custom `CategoryNav` above it. `src/theme/DocSidebar/CategoryNav.tsx` holds a **hardcoded `CATEGORIES` array** mapping category labels to doc routes (e.g. Installation → `/docs/tutorial-basics/create-a-document`). These links point at the scaffolding routes and must be updated by hand when real docs are added/renamed — they are not derived from the sidebar.
- **Live roadmap** — `src/pages/roadmap.tsx` renders `src/components/KanbanSection`, which fetches issues at runtime (client-side) from the **GitLab REST API** (`https://gitlab.com/api/v4/projects/sinhaparth5%2Fcoraza-waf-mod`). The project path is hardcoded in `KanbanSection/index.tsx`; update it there if the repo moves. Labels are color-coded by regex matching in `getLabelClass`.
- **Homepage** — `src/pages/index.tsx` plus `src/components/HomepageFeatures`.

### TypeScript

`tsconfig.json` extends `@docusaurus/tsconfig` with `strict: true`. It is only used by the IDE and `npm run typecheck` — Docusaurus itself uses Babel/SWC for compilation.

### Content authoring

- Docs support MDX (React components inside Markdown).
- Use `sidebar_position: N` frontmatter to control ordering within a directory.
- Broken internal links cause the build to throw (`onBrokenLinks: 'throw'`).
